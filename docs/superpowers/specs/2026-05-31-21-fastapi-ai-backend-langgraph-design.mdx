# FastAPI AI Backend with LangGraph + LangChain

## Context

SaaS Forge today has a working JavaScript AI layer in `packages/ai` (Vercel AI SDK, multi-provider) and a set of edge/server routes under `apps/web/app/api/ai/` (chat, TTS, STT). This layer is excellent for synchronous, streaming, in-request AI work and should stay.

What it does not cover well:

- Multi-step agentic workflows where the model decides what to do next (tool use, branching, conditional retries).
- Long-running graph executions that may take 30 seconds to many minutes and must survive page reload, network blip, or browser close.
- Retrieval-Augmented Generation (RAG) with first-class ingestion pipelines, vector stores, and reusable retriever abstractions.
- Workloads that need to run on a worker process independent of the Next.js request lifecycle.

The Python ecosystem (LangChain, LangGraph) is the dominant home for these patterns. Rather than reimplement a graph runtime and retriever framework in JavaScript, we introduce a Python service that handles only the workloads it is genuinely better at.

This spec specializes the more general work outlined in [`2026-04-07-06-background-jobs-task-queue-design.mdx`](./2026-04-07-06-background-jobs-task-queue-design.mdx) for the AI-specific path, and intentionally defers the browser-facing realtime push surface described in [`2026-04-07-18-real-time-infrastructure-sse-websockets-design.mdx`](./2026-04-07-18-real-time-infrastructure-sse-websockets-design.mdx) to Phase 2.

## Overview

Add `apps/backend/`, a FastAPI service written in Python 3.12, that runs:

- A streaming HTTP API for synchronous LangGraph agent executions.
- An ARQ worker (redis-backed async job queue) for long-running agentic and RAG jobs.
- A RAG layer with a swappable `VectorStore` (pgvector by default, Qdrant as alternate) and a `KnowledgeSource` abstraction agents compose.

The boundary is drawn by **workload shape, not by AI vs non-AI**:

- JS keeps: simple synchronous streaming chat, TTS/STT proxies, and any in-request AI work.
- Python takes: anything multi-step, graph-orchestrated, retrieval-heavy, or long-running.

Browsers never talk to Python directly. Web (Next.js) is the single trust boundary; it proxies streaming responses through to the browser and exposes a tRPC control plane for async jobs. Service-to-service auth between web and Python is HMAC-signed requests with a shared secret.

The backend ships as part of `templates/saas-boilerplate`, so scaffolded users get an AI-native stack out of the box.

## Main Approach

### Architectural decisions (locked)

| # | Decision | Choice | Rejected alternatives |
|---|---|---|---|
| 1 | Division between JS and Python | Split by workload shape | "Python handles all AI" (too disruptive); "Python is new default, JS untouched" (two stacks indefinitely) |
| 2 | Workload patterns supported | Both sync-streaming and async-job | Sync-only (insufficient for ingestion); async-only (kills interactive UX) |
| 3 | Redis provider | Upstash (TCP, persistent tier, NOT eviction) | Self-hosted (operational burden in prod); Upstash REST API (wrong fit for ARQ workers) |
| 4 | Public surface for backend | Internal-only; web proxies everything | Browser-direct with signed tokens (more attack surface for marginal latency gain) |
| 5 | Postgres access from Python | Reads everywhere via SQLAlchemy; writes only to Python-owned tables | Stateless Python via HTTP-only (too chatty); Python owns separate DB (lose cross-domain queries) |
| 6 | Vector store | pgvector default, Qdrant adapter (experimental in Phase 1), behind langchain's existing `VectorStore` interface | Pure pgvector (no escape hatch); Qdrant default (extra infra in template) |
| 7 | Job queue library | ARQ | Celery (sync-first, heavyweight); RQ (sync workers); Dramatiq (less ecosystem) |
| 8 | Python packaging | uv + pyproject.toml | Poetry (slower, weaker lockfile); pip-tools (no integrated runner) |
| 9 | ORM | SQLAlchemy 2.x async + asyncpg | Tortoise (smaller ecosystem); raw asyncpg (loses model layer) |
| 10 | Schema source of truth | Prisma owns all migrations; SQLAlchemy mirrors hand-written | Two migration tools (coordination hell); reflection only (loses type safety) |
| 11 | Auth between web ↔ backend | HMAC-SHA256 with shared secret + signed user context | JWT pass-through (entangles better-auth with backend); mTLS (overkill) |
| 12 | Job-status delivery to browser | tRPC polling at 1.5s, status hash in Upstash + terminal state in Postgres | SSE/WebSocket push (deferred to spec #18 Phase 2) |
| 13 | Template story | Ships in `templates/saas-boilerplate` | Root-only (loses "AI-native SaaS" positioning); scaffold-time toggle (added complexity) |

### Two execution flows

**Flow A — Sync-streaming agent.** Browser POSTs to `/api/ai/agents/:agentId/stream`. Web validates user via better-auth, builds an HMAC-signed payload, opens a streaming POST to `backend:8000/agents/stream`, pipes SSE events straight through to the browser (no buffering). Python runs the LangGraph and emits typed SSE events. Browser disconnect propagates as asyncio cancellation; LangGraph halts at the next `await`.

**Flow B — Async job.** Browser calls `trpc.aiJobs.create.mutate({ agentId, input })`. Web inserts an `AiJobRun` row in `PENDING` state, then HMAC-POSTs to `backend:8000/jobs` to enqueue. Returns `{ jobId }`. ARQ worker picks up, transitions to `RUNNING`, executes the LangGraph, writes events to `AiJobEvent` and live status to Upstash, finishes with `SUCCEEDED | FAILED | CANCELLED`. Browser polls `trpc.aiJobs.status.query({ jobId })` every 1.5s.

### API contracts

**Sync-streaming wire format (backend → web → browser SSE):**

| Event | Payload | Meaning |
|---|---|---|
| `step`        | `{ node, status: "start"\|"end", at }` | LangGraph node lifecycle |
| `token`       | `{ delta }`                            | LLM token stream chunk |
| `tool_call`   | `{ name, args, call_id }`              | Agent invoked a tool/retriever |
| `tool_result` | `{ call_id, result_summary }`          | Tool returned (summary; full via separate fetch) |
| `citation`    | `{ doc_id, chunk_id, score, snippet }` | RAG retrieval hit (for sources UI) |
| `final`       | `{ output: <agent JSON> }`             | Terminal payload |
| `error`       | `{ code, message }`                    | Stream-terminating error |
| `end`         | `{}`                                   | Last frame; client closes |

The wire format is our own narrow schema, not raw langchain `AIMessageChunk` objects, so the contract is stable across LangChain version bumps.

**Async job state machine:**

```
PENDING ──► RUNNING ──► SUCCEEDED
   │           │    └──► FAILED
   │           └──► CANCELLED  (cancel requested while running; honored at next node boundary)
   └─────────────► CANCELLED  (cancel requested before pickup)
```

**tRPC procedures (web side, new):**

| Procedure | Returns |
|---|---|
| `aiJobs.create({ agentId, input })`     | `{ jobId }` |
| `aiJobs.status({ jobId })`              | `{ id, status, agent_id, created_at, started_at?, finished_at?, latest_event?, result?, error? }` |
| `aiJobs.events({ jobId, sinceSeq?, limit? })` | `Array<{ seq, type, payload, at }>` (from `AiJobEvent`) |
| `aiJobs.cancel({ jobId })`              | `{ ok: true }` |

**Web-side status read order** (cheapest first):
1. `HGETALL job:{id}` from Upstash — sub-ms, has live progress (status hash refreshed with `EXPIRE 600` on every worker write; terminal-state hashes get `EXPIRE 300` and rely on Postgres after that).
2. On miss or terminal status, `SELECT … FROM "AiJobRun" WHERE id=$1`.

### Reliability mechanisms

**PENDING reaper.** ARQ worker runs at startup and on a 30s schedule:

```sql
SELECT id FROM "AiJobRun"
WHERE status = 'PENDING' AND created_at < now() - interval '60 seconds'
ORDER BY created_at LIMIT 100;
```

Each row is re-enqueued. ARQ deduplicates via `_job_id = ai_job_run.id`. After 5 minutes still `PENDING`, mark `FAILED` with `error_code = "ENQUEUE_TIMEOUT"`.

**Stale-heartbeat sweep.** ARQ worker on 60s schedule marks any `RUNNING` row whose `last_heartbeat_at > 5 minutes ago` as `FAILED` with `error_code = "STALE_HEARTBEAT"`. Worker writes `last_heartbeat_at` every ~30s during a job.

**Retries.** LLM transient errors (5xx, rate limit) retry inside the langchain call via `tenacity` (3 attempts, exponential backoff). ARQ task crashes retry once (`max_tries=2`). LLM non-retryable (4xx auth, invalid request) propagates as `FAILED`.

### RAG layer

**Concept hierarchy:**

```
Collection            ← user/org-owned logical container
  └── Document        ← something ingested (text blob, uploaded file)
        └── Chunk     ← split piece, embedded and retrieved
              └── Embedding   ← lives in the configured vector store
```

**Two Python abstractions:**

- `VectorStore` (low-level adapter). We use langchain's existing `VectorStore` interface, not a new one. Factory `get_vector_store(collection_id, embedder)` returns `PGVector` (default) or `QdrantVectorStore` based on `RAG_VECTOR_STORE` env.
- `KnowledgeSource` (high-level, what agents use). Binds a vector store + collection + retrieval policy. Method `async retrieve(query) -> list[RetrievedChunk]`. Emits `citation` SSE events.

**Embedding providers:**

| `RAG_EMBEDDER` env | Implementation | Notes |
|---|---|---|
| `openai:text-embedding-3-small` *(default)* | `OpenAIEmbeddings` | 1536 dims |
| `openai:text-embedding-3-large` | `OpenAIEmbeddings` | 3072 dims |
| `ollama:<model>` | `OllamaEmbeddings` | Local / offline |

Dimensions are locked per collection in `AiCollection.embeddingDims`; switching embedder requires re-ingestion. The `embedding` column is sized to the largest supported dimension (3072) with explicit `::vector(N)` casts at query time.

OpenRouter is intentionally NOT an embedder (it's a chat-completion router). It IS supported for chat via `langchain-openai` with `base_url="https://openrouter.ai/api/v1"`.

**Ingestion pipeline (worker-side, 5 named steps, each emits a `step` event):**

```
load    → fetch/read source bytes
extract → bytes → plain text
split   → text → chunks (RecursiveCharacterTextSplitter, 1000/200 default, override per input)
embed   → chunks → embeddings (batched with tenacity retries)
upsert  → (chunk_text + embedding) → Postgres + vector store
```

Chunk text **always** lives in Postgres (`AiDocumentChunk.text`), even when embeddings are in Qdrant. This makes re-indexing trivial: re-embed from `SELECT text FROM AiDocumentChunk` without needing the original source.

**Phase 1 source types:** `text` (raw content in payload) and `uploaded_file` (web uploads via existing storage flow, passes URL; backend supports `.txt`, `.md`, `.pdf` via `pypdf`). URL fetching, GitHub, Notion, Drive are Phase 2.

**Phase 1 canonical agent — `rag_chat`:**

```python
async def retrieve_node(state: RagChatState) -> dict:
    chunks = await state.knowledge.retrieve(state.query)
    return {"chunks": chunks}

async def answer_node(state: RagChatState) -> dict:
    answer = await llm_with_chunks(state.llm, state.query, state.chunks)
    return {"answer": answer}

def build_rag_chat_graph() -> StateGraph:
    g = StateGraph(RagChatState)
    g.add_node("retrieve", retrieve_node)
    g.add_node("answer", answer_node)
    g.add_edge(START, "retrieve")
    g.add_edge("retrieve", "answer")
    g.add_edge("answer", END)
    return g.compile()
```

Registered in `agents/registry.py` as `"rag_chat"`. Same graph drives both sync-stream and async-job entry points.

All additions live in `packages/database/prisma/ai.prisma` (or appended to existing files if multi-file isn't enabled — implementation plan decides). Migrations are NOT generated by automation per the repo's CLAUDE.md rule; the implementation plan will instruct the project owner to run `pnpm migrate`.

```prisma
enum AiJobStatus { PENDING RUNNING SUCCEEDED FAILED CANCELLED }

model AiJobRun {
```

**pgvector — hand-edited migration tail.** Prisma cannot express `vector(N)`. After `pnpm migrate` generates the migration, the operator appends:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
ALTER TABLE "AiDocumentChunk" ADD COLUMN "embedding" vector(3072);
CREATE INDEX "AiDocumentChunk_embedding_idx"
  ON "AiDocumentChunk" USING hnsw ("embedding" vector_cosine_ops);
```

This is the one place the design knowingly hand-edits SQL. The implementation plan documents it explicitly as a Prisma limitation.

**SQLAlchemy mirroring.** Hand-written async models in `apps/backend/src/saas_forge_backend/db/models.py` mirror the Prisma tables (same column names, types, indexes). A backend unit test introspects the live DB at startup and asserts model/schema agreement — prevents silent drift.

**Connection pooling caveat.** Python must connect directly to Postgres with its own pool (`asyncpg`), NOT through Prisma Accelerate. Accelerate is a JS-side proxy for serverless cold starts and adds a hop Python doesn't need.

### Auth implementation (HMAC)

- Algorithm: HMAC-SHA256.
- Secret: 32+ bytes hex/base64 in `BACKEND_HMAC_SECRET` env (set identically on both sides).
- Canonical body: `json.dumps(payload, sort_keys=True, separators=(",", ":"))`.
- Signed string: `f"{timestamp}\n{canonical_body}"`.
- Headers: `X-Saas-Forge-Ts`, `X-Saas-Forge-Sig`, `X-Saas-Forge-Req-Id`.
- Server rejects timestamps older than ±60s (replay defense). Optional Redis replay-set deferred to Phase 2.
- Rotation: set `BACKEND_HMAC_SECRET_NEXT`; backend accepts both for the rotation window.
- Signed payload always includes `{user_id, org_id, request_id}` which backend treats as proven.

Web-side helper lives in `packages/observability` as a thin signed-fetch wrapper, giving web ↔ backend calls consistent log context.

### Deployment

- **`Dockerfile.backend`** at repo root (mirrors `Dockerfile.web` placement). Multi-stage, two final targets: `api` and `worker` — same image, different `CMD`. Build context is the monorepo root so `apps/backend/` is at the expected path.
- **`docker-compose.yml`** gains:
  - `backend-api` service (`expose: ["8000"]`, no public port).
  - `backend-worker` service (no exposed ports).
  - `redis` graduates out of `profiles: [redis]` — it is now required.
  - `web` gains `BACKEND_URL=http://backend-api:8000` and `BACKEND_HMAC_SECRET` envs.
- **Production:** same Dockerfile, two services per environment. Recommended targets: Fly.io (two `fly.toml`), Railway (two services from one repo), Render. K8s users deploy two Deployments from one image. Documented in backend README.

### What is intentionally excluded from Phase 1

- Browsers calling backend directly (web always proxies).
- Python writing to user/auth/billing tables.
- Prisma running inside Python.
- Streaming async-job output (async = poll for terminal state; streaming is sync-only).
- Multi-region replication of jobs.
- A scaffold-time `--with-backend` toggle (every scaffold includes backend; rationale: "AI-native SaaS" positioning).

## Open Questions / Risks

- **Prisma schema folder mode**: spec assumes `prismaSchemaFolder` preview feature; implementation plan must verify and fall back to appending to an existing schema file if not enabled.
- **Vector dimension strategy**: Phase 1 uses a single `vector(3072)` column with explicit casts. If a deployment exclusively uses 1536-dim embeddings, the wider column wastes ~16KB per row. Acceptable for v1; Phase 2 may partition by dim.
- **Template scaffolding burden**: every scaffolded user now needs Python 3.12 + uv. If user feedback signals this is a barrier, a `--without-backend` scaffold flag becomes a Phase 1.5 add.
- **OpenRouter via langchain-openai**: relies on OpenRouter staying OpenAI-API-compatible; if they diverge meaningfully, a dedicated provider package will be needed.
